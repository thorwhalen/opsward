# Multi-Platform AI Agent Configuration: Formats, Emerging Standards, and Portability

## Abstract

Every coding-agent host invented its own way to carry project context — a memory file, a rules directory, a skills folder, a tool-protocol config — and the result is a thicket of overlapping, semantically-similar formats that drift out of sync. Two genuine convergence points have emerged in 2025–2026: **AGENTS.md** as a host-agnostic instruction file (now under the Linux Foundation's Agentic AI Foundation, 20,000+ repos) [1][2][3] and the **Model Context Protocol (MCP)** as the de-facto tool/data interop layer, with a federated registry and backing from Anthropic, OpenAI, Google, and Microsoft [10][11]. A third candidate, Anthropic's **Agent Skills (SKILL.md)**, was published as an open standard on 2025-12-18 and is being adopted as a portable capability unit across Codex, Gemini CLI, Copilot, and VS Code [7][8][9]. This report surveys the formats precisely, assesses what is and is not actually converging, and recommends a canonical single-source-of-truth strategy that composes the user's existing packages (`skill`, `aw`, `aw_agents`, `py2mcp`) and gives `opsward` ownership of the drift check that none of them currently provides.

## Scope

We cover four layers of agent configuration and their cross-host portability: (1) **instruction/memory files** (CLAUDE.md, AGENTS.md, GEMINI.md, copilot-instructions.md, CONVENTIONS.md, .cursorrules); (2) **scoped rules** (`.cursor/rules/*.mdc`, `.github/instructions/*.instructions.md`, `.windsurf/rules`, `.clinerules/`); (3) **portable capabilities** (Agent Skills / SKILL.md, subagents, prompt/command files); and (4) **tool/data protocols** (MCP, its `.mcp.json` host configs, and the registry). We treat hooks and IDE-proprietary settings as out of scope except where they bear on portability. Dates matter: this space changes monthly, so every claim is anchored to a dated source.

The table's most important reading is vertical: a single project that wants to satisfy Claude Code, Codex, Copilot, Cursor, Windsurf, Gemini, Aider, Cline, and Zed must today maintain — or generate — **nine instruction locations, four incompatible rule-scoping schemes, and N tool-protocol configs**. That fan-out is precisely the problem the convergence efforts target.

## The AGENTS.md convergence — real, but partial

AGENTS.md is the clearest convergence story. It is deliberately minimal: "a README for agents," just standard Markdown with no required fields — "use any headings you like" [1]. It emerged from collaboration across OpenAI Codex, Amp, Google Jules, Cursor, and Factory, and is now stewarded by the **Agentic AI Foundation under the Linux Foundation**, whose platinum backers include Anthropic, Google, Microsoft, and OpenAI [1][2]. Adoption is broad and measurable: InfoQ and BigGo report 20,000+ repositories carrying the file as of August 2025 [2][3], and the spec lists 20+ supporting agents including Codex, Jules, Gemini CLI, Copilot, Cursor, VS Code, Zed, Aider, goose, Factory, Devin, Windsurf, and JetBrains Junie [1].

**What it actually standardizes is narrow:** the file *name* and *location* (root, with nested files allowed where the closest one to the edited file wins) and a set of *conventional section topics* (setup commands, code style, testing) [1]. It does **not** standardize any schema, frontmatter, precedence rules across competing files, or behavior — by design. This is its strength (zero friction to adopt) and its weakness (it cannot express scoped/conditional rules, tool grants, or capabilities; those remain host-specific).

The sharpest tension is **AGENTS.md vs CLAUDE.md**. Claude Code does **not** read AGENTS.md natively — the official memory docs say it reads CLAUDE.md, and the "reads AGENTS.md as a fallback" claim circulating in issue threads is wrong [12]. The two reconciliation patterns that work are (a) a one-line `@AGENTS.md` import inside CLAUDE.md (Windows-safe, lets you keep Claude-only additions below the import) and (b) a filesystem symlink `ln -sf AGENTS.md CLAUDE.md` so they are literally one file (cleaner, but git symlinks need `core.symlinks = true` and can break on Windows) [12][20]. Either way the operator owns keeping them coherent — a gap, not a solved problem. Net assessment: **convergence is genuine at the "one plain-text brief per repo" layer and stalled everywhere richer.**

## MCP as the cross-platform tool/data protocol

Where instruction files converged on a *document*, tooling converged on a *protocol*. MCP turned one year old on 2025-11-25 and in that span went from an Anthropic experiment to the de-facto standard for connecting models to tools and data, with adoption by OpenAI (Agents SDK, Responses API, ChatGPT desktop, March 2025), Google DeepMind (Gemini, April 2025), and Microsoft — 97M+ monthly SDK downloads, 5,800+ servers, 300+ clients [11]. The decisive property for portability is **decoupling**: an MCP server is written once against the protocol and is consumable by *any* compliant host. The host-specific surface shrinks to a few lines of config (`.mcp.json`, `.cursor/mcp.json`, `~/.codex/config.toml`, `~/.gemini/settings.json`) pointing at the same server — the tool logic does not move.

The **MCP Registry** (preview 2025-09-08) closes the discovery half of the loop: a single source of truth at `registry.modelcontextprotocol.io` where maintainers self-publish once, with a **federation model** so public marketplaces and private enterprise sub-registries ingest, augment, or mirror the upstream via a shared OpenAPI schema [10][17][19][24]. The 2025-11-25 spec added an async **Tasks** primitive (call-now/fetch-later), CIMD client identity, and Cross-App Access for enterprise auth [11]. For a multi-platform configuration strategy, MCP is the layer to lean on hardest: it is the one place where "author once, run on every host" already holds with minimal translation loss, because the host config is a thin pointer rather than a re-expression of behavior.

## Agent Skills as a portable capability unit

If AGENTS.md standardizes the *brief* and MCP standardizes the *tools*, Agent Skills aims to standardize the *procedure* — a folder containing a `SKILL.md` (YAML frontmatter `name` + `description`, Markdown body, plus optional bundled `scripts/`, `references/`, `assets/`) governed by **progressive disclosure**: metadata pre-loaded into the system prompt, body loaded on relevance, bundled files pulled on demand [5][6][7]. Anthropic introduced Skills on 2025-10-16 and published them as an **open standard for cross-platform portability on 2025-12-18** [5], with a spec hosted at agentskills.io [7] and a public repo [8].

Cross-agent adoption is real but **conditionally portable**. As of 2026 the format is recognized by Claude Code, OpenAI Codex CLI, Gemini CLI, GitHub Copilot (via VS Code Agent Skills), Cursor (manual placement), and community ports for Cline, Windsurf, and OpenCode [9][25]. The core (folder + frontmatter + Markdown + progressive disclosure) transfers cleanly; the **advanced features do not** — Claude Code's `context: fork` subagent mechanism and Codex's `openai.yaml` metadata are host-specific and silently lost or ignored elsewhere [9]. So a "plain" skill is a strong portable unit today; a skill that leans on host-specific frontmatter is portable only after a lossy down-cast. This is the same lesson as AGENTS.md at the capability layer: **the lowest-common-denominator slice is portable; the expressive tail is not.**

## Portability strategies and failure modes

Three strategies are in production use, each with characteristic failure modes:

1. **Author-once-emit-many (transpilation).** A canonical source is compiled into each host's format. Tools like **rulesync** (`.rulesync/` → CLAUDE.md, `.cursorrules`, copilot-instructions, GEMINI.md, …, 20+ targets) [21], **ruler** ("apply the same rules to all coding agents") [22], and **skillport** ("manage once, serve anywhere" for Skills) [23] embody this. Failure mode: **lossy down-casts** — when the target format cannot express a source field (frontmatter, glob scoping, tool grants), the information is dropped, ideally with a warning, silently in the worst case.

2. **Symlinking a canonical file into per-host locations.** `ln -sf AGENTS.md CLAUDE.md` makes two conventions one file [20]. Failure modes: **no transformation** (works only when formats are byte-identical, so it cannot bridge CLAUDE.md→.mdc), Windows/git symlink fragility, and the inability to add host-only deltas without breaking the link.

3. **Generators + drift detection.** A generator writes per-host copies and a separate check verifies the rendered copies still match the canonical. This is the only strategy that survives **semantic drift** — the dominant real-world failure where someone hand-edits a generated CLAUDE.md, the canonical and the copy diverge, and different teammates get different agent behavior depending on which tool they opened [20]. Transpilers and symlinks *prevent* some drift; only an explicit drift check *detects* the drift that slips past them (manual edits, host-only additions, stale renders).

The cross-cutting failure mode behind all three is **the expressive tail**: host-specific features (Cursor activation modes, Copilot `applyTo` globs, Claude `context: fork`, Windsurf workflows) have no neutral representation, so any canonical format must choose between a thin lowest-common-denominator (portable, lossy) and a rich superset (expressive, requires per-host down-casting with documented loss).

## Grounding in the existing ecosystem

The user already owns most of the machinery; the gap is integration and drift detection, not green-field building.

- **`skill` (t/skill) — the closest existing multi-platform layer; COMPOSE.** Confirmed in source: `skill/install.py` defines an `agent_targets` registry mapping `claude-code → .claude/skills/{name}/SKILL.md`, `cursor → .cursor/rules/{name}.mdc`, `copilot → .github/copilot-instructions.md`; `skill/translate.py` implements `SKILL.md → .mdc`, `.mdc → Skill` (round-trip import), and `SKILL.md → copilot-instructions`, with a `translate(skill, *, target_format)` facade. Critically, the translators **already emit explicit lossy-translation warnings** ("Lossy translation to .mdc: dropping …") — i.e. `skill` has independently arrived at the expressive-tail problem and surfaces it. This is the SKILL-level transpiler to build on, not replace.
- **`aw` (t/aw) — the canonical IR pattern, scoped to agents; COMPOSE the pattern.** `aw/translators.py` already defines a normalized `AgentSpec` intermediate representation (`ToolSpec`, `ValidatorSpec`, `AgentSpec`) with `extract_agent_spec()` and author-once-emit-many renderers: `to_claude_skill` / `claude_skill_from_spec`, `write_skill_directory`, `to_crewai_yaml`, plus OpenAI tools+assistant emission. `aw/util.py:claude_desktop_config()` writes MCP server entries into `claude_desktop_config.json`. The *normalized-spec → many-targets* architecture is exactly the canonical SSOT shape we want — but `aw`'s IR describes *runtime agent objects*, not *project-level config files*. The lesson to reuse is the IR-with-renderers pattern, not the schema.
- **`aw_agents` (t/aw_agents) — multi-platform deploy; ADJACENT.** Ships `aw_agents/adapters/mcp.py` (MCP/Claude) and `scripts/deploy_api.py` (OpenAPI/ChatGPT) — deployment of running agents to hosts, complementary to config-file portability.
- **`py2mcp` (t/py2mcp) — feeds the MCP layer; COMPOSE.** Turns Python callables/stores/refs into an MCP server, i.e. the *producer* side of the one protocol (MCP) that already ports cleanly across hosts.
- **`opsward` (t/opsward) — already per-project and AGENTS.md-aware; EXTEND.** It generates CLAUDE.md, `.claude/skills`, and AGENTS.md (`--agents-md`, `_check_agents_md` consistency check) and scores cross-references — it is the only one of these tools whose job is *diagnosing and maintaining* config rather than producing a single artifact.

**The gap, precisely:** none of these owns a *host-agnostic canonical source of truth for project-level config that round-trips to every host*, plus *drift detection between the canonical and the rendered per-host copies*. `skill` translates SKILL units; `aw` normalizes agent runtimes; `opsward` checks one consistency pair (CLAUDE.md ↔ AGENTS.md). The missing piece is the connective tissue: a canonical representation + a drift check across all rendered copies.

## Recommendations

1. **Adopt a two-file canonical SSOT, not a new schema.** Make **AGENTS.md the canonical project brief** (it is the only instruction file with real cross-host adoption and Linux Foundation governance [1][2]) and **SKILL.md folders the canonical capability unit** (the only portable procedure format, now an open standard [5][7]). Resist inventing a bespoke superset format — the industry has voted, and a proprietary IR would itself become a drift source. CLAUDE.md becomes a thin shim that `@AGENTS.md`-imports the canonical brief plus Claude-only deltas [12].
2. **Render/symlink to hosts via `skill`'s translators — don't rebuild them.** Extend `skill`'s `agent_targets` registry to cover the remaining hosts (Windsurf `.windsurf/rules`, Gemini `GEMINI.md`, Cline `.clinerules/`, Codex nested AGENTS.md) and keep its lossy-warning discipline. For byte-identical pairs (AGENTS.md↔CLAUDE.md on non-Windows), prefer symlink; for format-divergent targets (.mdc, copilot-instructions, GEMINI.md), use transpilation.
3. **Lean on MCP for everything tool-shaped.** Treat MCP as the portable tool layer: produce servers with `py2mcp`, register them in the MCP Registry, and let each host's thin `.mcp.json`/settings pointer be the only per-host tool config. This is where author-once-run-everywhere already works with minimal loss [10][11].
4. **Let `opsward` own the drift check.** This is the load-bearing recommendation. Generalize `opsward`'s existing `_check_agents_md` into a **canonical↔rendered drift detector**: hash/normalize the canonical AGENTS.md and each SKILL.md, compare against every rendered per-host copy (.mdc, copilot-instructions, CLAUDE.md shim, GEMINI.md), and report (a) stale renders, (b) host-only hand edits that should be promoted to canonical, and (c) fields dropped in lossy down-casts. `opsward` is already the per-project diagnostician with cross-reference scoring — drift detection is the natural extension, and it is the gap no existing tool fills.
5. **Score portability as a first-class `opsward` dimension.** Add a "portability/coverage" signal: which hosts the repo currently satisfies, which canonical fields are non-portable (and to where), and whether renders are current — turning the abstract "is my config multi-platform?" question into a graded report.

## Open questions / decisions (seeds for sub-issues)

- **Canonical format: AGENTS.md alone, or AGENTS.md + a structured sidecar?** Plain Markdown cannot encode scoped/conditional rules or tool grants; do we accept lossy down-casts, or carry a small structured sidecar (YAML) that `opsward` owns and renders into per-host frontmatter?
- **Symlink vs transpile policy per target.** Which target pairs are safe to symlink (byte-identical) vs must be transpiled, and how do we degrade gracefully on Windows where git symlinks are fragile [12][20]?
- **Drift-check semantics.** Is the canonical authoritative (renders are throwaway, edits to renders are warnings) or is the relationship bidirectional (promote host-only edits back to canonical)? This decides whether the check is a *linter* or a *merge tool*.
- **Build-vs-adopt for transpilation.** Do we extend `skill`'s translators (in-ecosystem, lossy-aware) or shell out to `rulesync`/`ruler`/`skillport` [21][22][23]? Recommendation leans build-on-`skill`, but interop with `rulesync`'s `.rulesync/` could be a cheap on-ramp.
- **Where does the canonical live, and who triggers renders?** Repo-committed canonical + a pre-commit/CI render step, or on-demand `opsward generate`? Committing rendered copies makes them visible to every host but reintroduces drift risk.
- **Capability portability boundary.** Do we forbid host-specific SKILL.md frontmatter (`context: fork`, `openai.yaml`) in canonical skills to guarantee portability, or allow it and down-cast with documented loss [9]?
- **MCP config unification.** Is there value in a single canonical MCP server list rendered into each host's pointer config, or is that fan-out already thin enough to ignore?
